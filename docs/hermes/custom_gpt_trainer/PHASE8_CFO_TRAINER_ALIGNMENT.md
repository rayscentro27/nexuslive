# Phase 8: Custom GPT Trainer Alignment
**Date:** 2026-06-03  
**Scope:** How the Custom GPT Trainer uses Phase 8 artifacts

---

## Role of the Custom GPT Trainer

The Custom GPT Trainer is NOT the live Hermes brain. It is the **evaluator, coach, and prompt/test generator** that:

1. Reviews failed Telegram transcripts
2. Compares actual vs. expected responses
3. Generates improved prompts and test cases
4. Keeps Hermes aligned with the CFO doctrine
5. Produces training examples from failure traces

---

## What It Uses

### 1. Failed Telegram Transcripts

Source: `prototypes/fixtures/phase8_failed_telegram_cases.json`

Each case includes:
- The raw user message (exactly as typed in Telegram)
- The context at the time (state fields)
- The expected behavior (intent, tool, required/forbidden phrases)

The Trainer reads these and asks: "What was wrong about the actual response, and what would the ideal CFO response look like?"

**Usage:** Generate rewrite examples. Each rewrite becomes a training pair: (bad response, ideal response).

---

### 2. Phase 8 Traces

Source: `docs/reports/strategy/phase8_cfo_loop_traces.jsonl`

Each trace entry includes:
- Message + intent + confidence
- Retrieved evidence keys
- Tool selected + tool args
- Tool result summary
- Final response header
- Safety notes

**Usage:** The Trainer reviews traces where confidence < 0.8 or where the intent was "unknown_answer" — these are candidates for improved prompts or new intent patterns.

---

### 3. Tool Schemas

Source: `docs/reports/strategy/phase8_hermes_tool_schemas.json`

The Trainer uses tool schemas to:
- Verify that Hermes responses are consistent with tool behavior
- Generate test assertions for each tool
- Flag responses that describe actions a tool should not take

**Example:** If the Trainer sees a response that mentions "published to subscribers" for a `create_implementation_prompt` call, it flags this as a safety violation.

---

### 4. Doctrine Files

Source: `docs/hermes/` directory

Relevant doctrine:
- `HERMES_CFO_CONVERSATION_CONTRACT.md` — conversation structure rules
- `HERMES_PLAIN_LANGUAGE_STYLE_GUIDE.md` — response format expectations
- `HERMES_FAILURE_LEARNING_PROTOCOL.md` — how failures are processed
- `HERMES_UNKNOWN_ANSWER_PROTOCOL.md` — fallback behavior rules

**Usage:** The Trainer checks every rewrite against the doctrine. A rewrite that violates the style guide is rejected.

---

### 5. Expected Rewrites

For each failed case, the Trainer produces:

```json
{
  "case_id": "case_01",
  "original_response": "...",
  "failure_reason": "Showed daily plan instead of draft comparison",
  "ideal_response": "PLAIN ANSWER\n\nHere is what changed...",
  "doctrine_reference": "HERMES_CFO_CONVERSATION_CONTRACT.md §3"
}
```

These rewrites are stored in `prototypes/fixtures/phase8_rewrites.json` (generated by Trainer).

---

### 6. Generated Tests

The Trainer produces test assertions that can be added to `scripts/test_phase8_cfo_loop_evaluation_cases.py`:

```python
check("case_01: response contains 'changed'", "changed" in response.lower())
check("case_01: response does not contain 'daily plan'", "daily plan" not in response.lower())
```

These tests are runnable immediately — no manual review needed.

---

## Trainer Workflow

```
Failed transcript → Trainer reads context + expected + actual
                 → Identifies root cause (wrong intent / wrong tool / wrong format)
                 → Writes rewrite (ideal response)
                 → Checks rewrite against doctrine
                 → Generates test assertions
                 → Outputs fixture update (new case or updated case)
                 → Outputs new test assertions
```

---

## What the Trainer Does NOT Do

- Does NOT run as the live Hermes bot
- Does NOT call the Telegram API
- Does NOT modify production routing
- Does NOT execute tools
- Does NOT approve, publish, spend, or deploy
- Does NOT access Supabase directly

The Trainer's output is **text artifacts** (rewrites, test cases, prompt improvements) that a developer (Claude Code) then applies.

---

## Integration with Phase 8 Loop

The weekly improvement cycle:

1. **Monday**: Telegram logs reviewed → failures extracted to fixtures
2. **Tuesday**: Trainer generates rewrites + test assertions
3. **Wednesday**: Rewrites reviewed by Ray → approved rewrites applied
4. **Thursday**: Updated tests run → pass/fail tracked
5. **Friday**: Evaluation report generated → delta vs. previous week

This creates a continuous improvement loop without requiring manual phrase-by-phrase patches.
